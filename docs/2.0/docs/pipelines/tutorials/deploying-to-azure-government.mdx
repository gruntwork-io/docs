# Connecting Pipelines to Azure Government

In this tutorial, we will guide you through configuring Pipelines to authenticate against an Azure Government subscription and run plan/apply against the Azure Government cloud endpoints.

## What you'll get

By the end of this tutorial, you will have:

- A Pipelines configuration that targets an Azure Government subscription.
- A `root.hcl` provider block that points the azurerm and azuread providers at the Azure Government cloud.
- A vended set of files generated by Boilerplate that wires the `cloud_environment` value through to both Pipelines and the Terragrunt provider blocks.

## Prerequisites

Before starting, ensure you have the following:

- Pipelines installed in a GitHub or GitLab repository. Refer to [Setup & Installation](/2.0/docs/pipelines/installation/overview) for more details.
- Familiarity with Pipelines [Configurations as Code](/2.0/reference/pipelines/configurations-as-code) and the [Authenticating to Azure](/2.0/docs/pipelines/concepts/cloud-auth/azure) concept page.
- Access to an Azure Government subscription and Entra ID tenant.
- Permissions in that Entra ID tenant to create app registrations and federated identity credentials.
- Permissions to open a pull request or merge request in the GitHub/GitLab repository where Pipelines is installed.
- Boilerplate installed on your development machine. If you have [mise](https://mise.jdx.dev/getting-started.html) installed that is as simple as `mise use boilerplate`, alternatively you can download it from the [release page](https://github.com/gruntwork-io/boilerplate/releases/).

:::info

This tutorial should take approximately 30 minutes to complete.

:::

### Necessary Configuration Values

- <CustomizableValue id="SUBSCRIPTION_NAME"/>: The name of the Azure Government subscription. This name will be used as the environment name in Pipelines configuration and as the folder name in your repository.
- <CustomizableValue id="SUBSCRIPTION_ID"/>: The Azure Government subscription ID.
- <CustomizableValue id="TENANT_ID"/>: The Azure Government Entra ID tenant ID.
- <CustomizableValue id="PLAN_CLIENT_ID"/>: The Entra ID application (client) ID that Pipelines will authenticate as when running `terragrunt plan`. Typically scoped to read-only permissions.
- <CustomizableValue id="APPLY_CLIENT_ID"/>: The Entra ID application (client) ID that Pipelines will authenticate as when running `terragrunt apply`. Typically scoped to read/write permissions.

## Overview

Pipelines targets the public Azure cloud by default. Azure Government uses a distinct set of endpoints (`login.microsoftonline.us` for authentication, `*.usgovcloudapi.net` for management), so connecting Pipelines to it requires telling each layer of the stack which Azure cloud to use.

There are three places where the Azure cloud environment must be set for a working Azure Government setup:

1. **Entra ID application registrations**: created in the [Azure Government portal](https://portal.azure.us), not the public Azure portal.
2. **Pipelines HCL**: `cloud_environment = "usgovernment"` on the `azure_oidc` block, so Pipelines exports `ARM_ENVIRONMENT=usgovernment` to Terragrunt. The accepted values match the azurerm provider exactly (lowercase): `public`, `usgovernment`, `china`, `german`, `stack`.
3. **Generated provider blocks**: `environment = "usgovernment"` on both the `azurerm` and `azuread` provider blocks (typically generated by your `root.hcl`), so local Terragrunt runs target the same cloud as Pipelines.

## Setting up Entra ID Applications in Azure Government

All Entra ID work for an Azure Government subscription must happen in the [Azure Government portal](https://portal.azure.us). Application registrations created in the public portal (`portal.azure.com`) cannot federate to an Azure Government subscription.

1. Sign in to [portal.azure.us](https://portal.azure.us) with an account that has permission to register applications in the Azure Government tenant.
2. Create two app registrations: one that will be used for plan (read-only) and one for apply (read/write). Note each application's **Application (client) ID** for use later.
3. For each app registration, add a **Federated credential** that trusts your CI provider:

<TabItem value="github" label="GitHub" default>

- **Federated credential scenario**: GitHub Actions deploying Azure resources.
- **Issuer**: `https://token.actions.githubusercontent.com`
- **Subject identifier**: `repo:$$GITHUB_ORGANIZATION$$/$$GITHUB_REPOSITORY$$:ref:refs/heads/main` (for apply) or a pull-request-scoped subject (for plan). Refer to the [GitHub docs on OIDC subject claims](https://docs.github.com/en/actions/deployment/security-hardening-your-deployments/about-security-hardening-with-openid-connect) for the full set of available formats.
- **Audience**: `api://AzureADTokenExchange`

</TabItem>
<TabItem value="gitlab" label="GitLab">

- **Federated credential scenario**: Other / OpenID Connect.
- **Issuer**: your GitLab instance's OIDC issuer (`https://gitlab.com` for GitLab.com).
- **Subject identifier**: a subject that matches the GitLab CI jobs Pipelines runs. Refer to the [GitLab docs on OIDC subject claims](https://docs.gitlab.com/ee/ci/secrets/id_token_authentication.html).
- **Audience**: `api://AzureADTokenExchange`

</TabItem>
</Tabs>

4. Grant each application the appropriate role assignments on the Azure Government subscription. A common starting point is `Reader` for the plan application and `Contributor` for the apply application; refine as needed.

## Updating the Pipelines HCL config

Add a `cloud_environment = "usgovernment"` attribute to the `azure_oidc` block for the Azure Government environment:

```hcl title=".gruntwork/environments.hcl"
environment "$$SUBSCRIPTION_NAME$$" {
  filter {
    paths = ["$$SUBSCRIPTION_NAME$$/*"]
  }

  authentication {
    azure_oidc {
      tenant_id         = "$$TENANT_ID$$"
      subscription_id   = "$$SUBSCRIPTION_ID$$"
      plan_client_id    = "$$PLAN_CLIENT_ID$$"
      apply_client_id   = "$$APPLY_CLIENT_ID$$"
      cloud_environment = "usgovernment"
    }
  }
}
```

With this in place, when Pipelines authenticates for a unit under `$$SUBSCRIPTION_NAME$$/`, it will export `ARM_ENVIRONMENT=usgovernment` to Terragrunt alongside the other `ARM_*` variables, and the azurerm provider will hit the Azure Government endpoints.

## Updating the provider block

Pipelines exporting `ARM_ENVIRONMENT` is enough for `terragrunt plan`/`apply` invoked via Pipelines to work. To keep local Terragrunt runs targeting the same cloud, pass the same value through to the generated `azurerm` and `azuread` provider blocks in your `root.hcl`:

```hcl title="root.hcl"
generate "providers" {
  path      = "providers.tf"
  if_exists = "overwrite_terragrunt"
  contents  = <<EOF
provider "azurerm" {
  features {}

  environment                     = "usgovernment"
  resource_provider_registrations = "none"
}

provider "azuread" {
  environment = "usgovernment"
}
EOF
}
```

The OpenTofu/Terraform state backend (`backend "azurerm"`) must also be configured for Azure Government — typically a storage account in an Azure Government region with its `environment` attribute set to `usgovernment`.

## Using Boilerplate

The [`terragrunt-scale-catalog`](https://github.com/gruntwork-io/terragrunt-scale-catalog) Azure subscription templates accept a `CloudEnvironment` variable that is threaded into both the Pipelines HCL and the generated provider blocks. To vend a new Azure Government subscription:

1. Create a `vars.yaml` file on your local machine with the following content:

<Tabs groupId="platform">
<TabItem value="github" label="GitHub" default>
    ```yaml title="vars.yaml"
    SubscriptionName: "$$SUBSCRIPTION_NAME$$"
    SubscriptionId: "$$SUBSCRIPTION_ID$$"
    TenantId: "$$TENANT_ID$$"
    PlanClientId: "$$PLAN_CLIENT_ID$$"
    ApplyClientId: "$$APPLY_CLIENT_ID$$"
    CloudEnvironment: usgovernment
    ```
</TabItem>
<TabItem value="gitlab" label="GitLab">
    ```yaml title="vars.yaml"
    SubscriptionName: "$$SUBSCRIPTION_NAME$$"
    SubscriptionId: "$$SUBSCRIPTION_ID$$"
    TenantId: "$$TENANT_ID$$"
    PlanClientId: "$$PLAN_CLIENT_ID$$"
    ApplyClientId: "$$APPLY_CLIENT_ID$$"
    CloudEnvironment: usgovernment
    ```
</TabItem>
</Tabs>

2. From the root of your repository, run Boilerplate against the Azure subscription template:

<Tabs groupId="platform">
<TabItem value="github" label="GitHub">
```bash
boilerplate --template-url "git@github.com:gruntwork-io/terragrunt-scale-catalog.git//templates/boilerplate/azure/github/subscription?ref=X.Y.Z" --output-folder . --var-file vars.yaml --non-interactive
```
</TabItem>
<TabItem value="gitlab" label="GitLab">
```bash
boilerplate --template-url "git@github.com:gruntwork-io/terragrunt-scale-catalog.git//templates/boilerplate/azure/gitlab/subscription?ref=X.Y.Z" --output-folder . --var-file vars.yaml --non-interactive
```
</TabItem>
</Tabs>

Boilerplate generates `.gruntwork/environment-$$SUBSCRIPTION_NAME$$.hcl`, a `root.hcl` with the matching provider blocks, and a bootstrap stack for the subscription.

## Testing

1. Point the Azure CLI at Azure Government on your local machine so you can validate access:

   ```bash
   az cloud set --name AzureUSGovernment
   az login
   az account show
   ```

2. Commit the generated files and open a pull request or merge request. Pipelines will run `terragrunt plan` against the Azure Government endpoints. Watch the Pipelines logs for the value of `ARM_ENVIRONMENT` (logged as `cloud_environment`) and confirm requests are hitting `*.usgovcloudapi.net`.

3. Merge the PR/MR. Pipelines will run `terragrunt apply` against Azure Government.

## Troubleshooting

| Symptom | Likely cause | Fix |
| --- | --- | --- |
| `AADSTS70021: No matching federated identity record found` | Federated identity credential created in the public portal or scoped to the wrong repo/branch. | Recreate the federated credential in [portal.azure.us](https://portal.azure.us) with the correct subject identifier. |
| Pipelines plan succeeds but `terragrunt plan` fails locally with `Error building ARM Config: Authenticating using the Azure CLI is only supported as a User`. | Local Azure CLI is logged into the public Azure cloud, but `root.hcl` selects Azure Government. | Run `az cloud set --name AzureUSGovernment && az login` before running Terragrunt locally. |
| `cloud_environment must be one of [public usgovernment china german stack]` at parse time. | Misspelled or uppercased value (for example `USGovernment`). | The azurerm provider requires lowercase. Use `usgovernment`. |
| `terragrunt apply` works under Pipelines but local runs hit public Azure endpoints. | `azurerm` / `azuread` provider blocks in `root.hcl` are missing `environment = "usgovernment"`. | Regenerate `root.hcl` from the boilerplate template with `CloudEnvironment: usgovernment`, or add `environment = "usgovernment"` manually. |
| State backend errors like `The specified resource does not exist` or DNS failures to `*.blob.core.windows.net`. | The state backend storage account is in Azure Government but `backend "azurerm"` is pointed at the public-cloud endpoint. | Configure your `backend "azurerm"` block for Azure Government (the storage account's fully qualified hostname will end in `*.blob.core.usgovcloudapi.net`). |
